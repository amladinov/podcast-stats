# Import Format Strategy

Дата: 2026-04-03  
Проект: `podcast-stats`  
Статус: active reference

## Зачем нужен этот документ

Этот документ фиксирует продуктовую и логическую модель импорта данных в `podcast-stats`.

Он нужен как source of truth для:

- текущей реализации импорта
- дальнейшей разработки `Mave`
- возможной будущей интеграции с `Mave`
- восстановления логики проекта при продаже, передаче или пересборке сервиса

## Product Principle

У сервиса должно быть не одно, а несколько качественно разных способов импорта.

Иерархия такая:

1. `CSV / API import`
2. `Paste import`

Это не равнозначные форматы. У каждого свой уровень точности, стоимости разработки и смысл данных.

`Screenshot / OCR` пока не входит в пользовательскую продуктовую иерархию. Это отдельное исследовательское направление, а не обещанный способ импорта.

## Каноническая модель форматов

### 1. CSV / API import

Это основной и предпочтительный путь.

Использовать, когда платформа даёт:

- официальный CSV
- API
- OAuth-backed data access

Что даёт:

- максимальную точность
- предсказуемую схему
- корректную историческую динамику
- меньше ручной валидации

В продукте это должен быть `primary import`.

### 2. Paste import

Это альтернативный путь, когда пользователь не может выгрузить CSV, но может скопировать структурированный текст из интерфейса платформы.

Что даёт:

- дешёвый fallback
- хорошую точность, если структура текста стабильна
- быстрый UX без OCR

Что не даёт:

- не всегда даёт полноценную историю
- может давать только snapshot на дату копирования

В продукте это должен быть `secondary import`, но официальный и поддерживаемый.

### 3. Screenshot / OCR import

Это потенциальный исследовательский путь для случаев, когда:

- нет CSV
- нет API
- копирование текста неудобно или невозможно

Что даёт:

- максимальную доступность для пользователя

Что усложняет:

- extraction pipeline
- confidence scoring
- ручную проверку
- поддержку разных экранов и layout

В продукте это пока не публичный способ импорта. Это research track, а не обещанная часть интерфейса.

## Семантика данных

Нужно жёстко разделять два типа импортов по смыслу данных:

### Historical import

Это импорт, который описывает значения за период и подходит для динамики.

Типичные признаки:

- есть диапазон дат
- есть временные точки
- данные можно честно положить в timeline

К этой категории обычно относятся:

- `CSV`
- `API`

### Snapshot import

Это импорт, который описывает актуальные накопленные значения на момент загрузки.

Типичные признаки:

- есть текущие totals
- нет надёжной временной разбивки
- это снимок состояния “на сегодня”

К этой категории обычно относятся:

- `Paste import`
- часть возможных будущих vision / OCR imports

## Главное правило продукта

Нельзя смешивать `historical` и `snapshot` так, будто это два одинаковых числовых слоя.

Иначе сервис начинает:

- дважды считать одни и те же данные
- строить фальшивую динамику
- искажать totals

## Текущая логика Mave

Для `Mave` уже зафиксирована гибридная модель:

- `Mave CSV` = источник исторической динамики
- `Mave Paste Snapshot` = источник актуальных totals по эпизодам

### Что это значит на практике

Если у пользователя есть только `Mave Paste`:

- сервис может показать актуальные totals
- сервис не должен выдумывать полноценную временную динамику Mave

Если у пользователя есть `Mave CSV`, а потом он загружает `Mave Paste`:

- `CSV` остаётся источником timeline
- `Paste` обновляет актуальные totals
- данные не суммируются как два независимых импорта

Если позже пользователь снова загружает новый `Mave CSV`:

- `CSV` снова становится авторитетным источником и для timeline, и для totals
- старый paste-snapshot не должен дублировать эти же цифры поверх CSV

## Правило авторитетности источников

Для каждой платформы у сервиса должна быть явная иерархия источников.

### Текущее правило

1. Если есть официальный `CSV / API`, он приоритетнее по историческим данным.
2. Если есть `Paste Snapshot`, он может обновлять только текущие totals.
3. Если когда-то появится `Screenshot import`, он по умолчанию должен вести себя как `snapshot`, пока не доказано обратное.

## Правило UI

Пользователь должен понимать не только “что он загрузил”, но и “что это значит”.

Поэтому в интерфейсе важно различать:

- `CSV`
- `Paste Snapshot`

И отдельно пояснять, что именно даёт каждый способ:

- `CSV / API` = историческая динамика
- `Paste` = актуальные значения на дату импорта

## Рекомендуемый порядок в интерфейсе

Если у платформы есть несколько пользовательских способов загрузки, их нужно показывать в таком порядке:

1. `Загрузить CSV` или `Подключить API`
2. `Вставить скопированный текст`

Почему именно так:

- это честно отражает качество источников
- помогает пользователю сначала выбрать самый точный путь
- снижает количество поддерживаемых ошибок

## Mave как продуктовый кейс

`Mave` стал первой платформой, где это разделение пришлось формализовать.

Почему:

- CSV доступен не всем
- текст из интерфейса копируется достаточно стабильно
- screenshot/OCR остаётся возможным, но более дорогим следующим шагом

### Зафиксированная стратегия для Mave

1. Основной путь: `Mave CSV`
2. Альтернативный путь: `Mave Paste Snapshot`

## Что считать canonical truth

Для сервиса canonical truth должен определяться не только платформой, но и типом импорта.

### Canonical truth по слоям

- `timeline truth` берётся из `historical import`
- `current totals truth` берётся из самого свежего валидного источника totals

Это значит:

- один и тот же источник не обязан быть каноническим для всего сразу
- допустима гибридная модель, если она прозрачна и не приводит к двойному суммированию

## Чего нельзя делать

- Нельзя автоматически суммировать `CSV` и `Paste`, если они описывают одну и ту же платформу.
- Нельзя строить график исторической динамики из чистого snapshot-импорта.
- Нельзя скрывать от пользователя, что данные пришли из fallback-источника.
- Нельзя считать screenshot-import равным CSV по качеству без validation layer.

## Когда нужен preview

Preview обязателен для:

- `Paste import`

Preview не обязателен в том же объёме для:

- чистого `CSV`
- контролируемого `API`

### Что должен показывать preview

- сколько записей распознано
- какие поля извлечены
- что не распарсилось
- какие значения требуют проверки

## Future-facing правило для screenshot/OCR

Если в будущем добавляется `Screenshot Import`, по умолчанию он должен проектироваться как:

- `fallback import`
- `snapshot-first import`
- `validation-required import`

То есть сначала:

- распознаём
- показываем confidence / preview
- даём исправить руками
- только потом импортируем

## Short Operating Model

Короткая операционная модель сервиса:

1. Сначала ищем официальный структурированный источник данных.
2. Если его нет, пробуем текстовый импорт.
3. Historical и snapshot всегда разделяем логически.
4. Источники одной платформы не суммируем автоматически без понимания их семантики.

## Current Decision Snapshot

На дату этого документа решение зафиксировано так:

- `CSV / API` остаётся главным форматом импорта
- `Paste` является официальным альтернативным способом импорта
- `Screenshot / OCR` остаётся внутренним исследовательским направлением и не обещается в интерфейсе
- `Mave Paste Snapshot` уже реализован как рабочий fallback
- приоритет качества данных выше, чем количество поддерживаемых способов загрузки
