# Mave Screenshot Import Tasks

Дата: 2026-04-01
Проект: `podcast-stats`
Статус: backlog / future work

## Контекст

Сейчас сервис умеет хорошо работать с CSV-выгрузками Mave, но для их получения нужна платная подписка Mave+.

Практическая проблема:

- большинство тестовых пользователей не будут оплачивать подписку только ради экспорта
- значит нужен альтернативный путь импорта данных

Цель:

- дать пользователю возможность загрузить набор скриншотов из личного кабинета Mave
- извлечь из них данные
- превратить извлечённые значения в наш собственный CSV / internal structured format
- затем использовать этот же формат для построения аналитики внутри `podcast-stats`

## Product Goal

Нужен поток:

1. пользователь загружает скриншоты из Mave
2. сервис распознаёт данные со скриншотов
3. сервис собирает структурированные записи по эпизодам и датам
4. сервис формирует собственный CSV или JSON-формат
5. дальше данные идут в существующий pipeline аналитики

## Key Unknowns

Перед реализацией нужно зафиксировать:

1. Какие именно экраны Mave пользователь будет скринить:
   - общая статистика по эпизодам
   - таблица по дням
   - графики
   - карточка конкретного эпизода
2. Достаточно ли таблиц/карточек, или придётся извлекать цифры с графиков
3. Один скриншот покрывает один эпизод или один временной диапазон
4. Нужно ли поддерживать мобильные и десктопные скриншоты одновременно
5. Нужна ли полуавтоматическая ручная проверка распознанных данных

## Backlog

### Phase 1. Исследование источника

1. Собрать 10-20 реальных скриншотов Mave разных типов
2. Классифицировать шаблоны экранов
3. Зафиксировать, какие данные можно стабильно снимать OCR, а какие нет
4. Определить минимальный набор экранов, достаточный для построения полезной аналитики
5. Проверить, есть ли на скриншотах таблица с текстом или только графическое представление

### Phase 2. Спроектировать внутренний формат

1. Описать intermediate schema для распознанных данных из Mave screenshots
2. Решить, что будет canonical-форматом:
   - собственный CSV
   - внутренний JSON
   - оба варианта
3. Определить обязательные поля:
   - episodeTitle
   - date или month
   - plays
   - source screenshot id
   - confidence score
4. Определить, как хранить частично распознанные строки
5. Определить, как помечать конфликтующие или сомнительные значения

### Phase 3. Extraction pipeline

1. Сделать загрузку изображений в интерфейсе
2. Добавить preprocessing:
   - crop
   - contrast
   - grayscale
   - deskew при необходимости
3. Реализовать OCR/vision extraction
4. Добавить layout parsing:
   - поиск заголовков колонок
   - поиск названий эпизодов
   - поиск дат
   - поиск числовых значений
5. Собрать parser, который превращает OCR output в structured rows

### Phase 4. Validation layer

1. Добавить confidence scoring для каждой строки
2. Сделать экран ручной проверки распознанных значений
3. Подсвечивать:
   - пропущенные эпизоды
   - дубли
   - подозрительные скачки значений
   - несовпадение дат
4. Дать возможность быстро исправить данные руками перед импортом
5. Разрешить экспорт промежуточного CSV для отладки

### Phase 5. Integration with existing analytics

1. Привязать screenshot-import к текущему Mave parser flow
2. Либо:
   - генерировать CSV в формате, который уже понимает `parseMave`
3. Либо:
   - сделать отдельный parser `parseMaveScreenshots`
4. Подключить это к текущему `uploadPlays`
5. Проверить, что итоговые данные корректно отображаются в dashboard

### Phase 6. UX

1. Добавить новый способ импорта на setup-странице:
   - `CSV`
   - `Скриншоты Mave`
2. Объяснить пользователю, какие скриншоты нужны
3. Добавить примеры правильных скриншотов
4. Показать прогресс распознавания
5. Показать статус:
   - распознано автоматически
   - требует проверки
   - готово к импорту

## Technical Directions To Evaluate

### Option A. OCR-first

Идея:

- обычный OCR + собственный parser таблиц

Плюсы:

- дешевле
- проще контролировать pipeline

Минусы:

- хуже работает на сложных layout
- хрупко при изменении UI Mave

### Option B. Vision model extraction

Идея:

- использовать vision model, которая сразу возвращает structured JSON

Плюсы:

- лучше понимает layout
- проще стартовать на нестандартных скриншотах

Минусы:

- дороже
- нужен контроль качества и повторяемости

### Option C. Hybrid

Идея:

- OCR для простых таблиц
- vision fallback для сложных экранов

Плюсы:

- лучший баланс качества и стоимости

Минусы:

- более сложная архитектура

## Recommended Initial MVP

Самый реалистичный MVP:

1. Поддерживать только десктопные скриншоты
2. Поддерживать только один-два конкретных типа экранов Mave
3. Не распознавать графики, только таблицы/карточки с текстовыми числами
4. После распознавания обязательно показывать ручную проверку
5. На выходе генерировать внутренний CSV, совместимый с уже существующим Mave import flow

## Concrete Next Tasks

Если начинать реализацию, идти в таком порядке:

1. Собрать референсный набор скриншотов Mave
2. Зафиксировать, какой именно экран идёт в MVP
3. Описать target schema для extracted rows
4. Сделать отдельный draft parser pipeline вне основного UI
5. После первого точного extraction prototype встроить это в setup flow

## Definition of Success

Решение можно считать успешным, если:

1. пользователь без платной подписки Mave может импортировать данные со скриншотов
2. результат даёт пригодную аналитику в dashboard
3. ошибки распознавания можно быстро исправить вручную
4. MVP работает хотя бы на одном стандартизованном сценарии скриншотов
